## 幻灯片 1｜13 天，三个人，一套生产级竞拍系统

大家好。我们今天要展示的，是一套面向直播电商场景的实时竞拍系统——支持高并发出价、多端实时同步、弱网恢复，以及从商品发布到订单成交的完整链路。

这套系统我们三个人用了 13 天完成。我想先说一下为什么 13 天这个数字值得关注。

一个覆盖后端 Go 服务、移动端 H5、PC 商家后台的全栈系统，如果按照传统开发流程，光是接口对齐和联调就要反复沟通两三周。我们用了一套基于 AI 的协作方法，把这个时间压到了 13 天。接下来我先讲这套方法，再讲系统本身。

---

## 幻灯片 2｜合同先行 × Mock 优先 × 角色专精

我们的方法论可以用三个词概括：**合同先行、mock 优先、角色专精**。

第一步，在动任何代码之前，我们用 AI 协作把所有的接口合同锁死——包括 REST 接口定义、WebSocket 事件格式、数据库 schema、错误码体系。这份合同（contract-v2.md）是三个人并行开发的地基。合同锁定后，任何人修改都需要三人 review 并写 changelog，不允许私自改动。

第二步，有了合同，前端不需要等后端。我们让前端用 MSW（Mock Service Worker）直接拦截请求，按照 openapi.yaml 返回 mock 数据；后端开 `MOCK_MODE` 让未完成的模块不阻塞已完成的模块。**第一天，三个人就真正并行了。**

第三步，后端某个模块完成后，@前端负责人，前端删掉对应的 MSW handler，切真接口，验收通过才算完成。mock 和真接口同时存在不超过 24 小时，防止漂移。

这套流程的核心是：把不确定性前移到合同阶段解决，开发阶段只是执行。

---

## 幻灯片 3｜AI 做什么，人做什么

很多人听到"AI 辅助开发"会问：那代码是你们写的还是 AI 写的？

这个问题问反了。更重要的是：**哪些决策是 AI 没法替你做的？**

AI 承担了大量重复性的脑力工作：生成 openapi.yaml 初稿、MSW handler 代码、标准 CRUD、错误码映射文件、压测脚本、文档整理。这些东西不需要创造力，但非常费时间。

而我们把认知资源集中在 AI 无法替代的地方：出价一致性的架构决策——要几道防线、防线的顺序是什么；跨团队依赖的协调——谁等谁、等多久切 mock；联调时 bug 的根因分析；以及性能 tradeoff 的判断。

简单说：**AI 帮我们把能自动化的重复思考变成 0，让我们可以把时间花在真正需要判断的地方。**

## 幻灯片 4｜13 天里程碑节奏

这是我们 13 天的节奏安排。五个阶段，每个阶段有明确的出口标准。

D1-D2 是"底座"阶段，目标是让三人并行解锁：docker 环境跑通、登录桩接口可用、前端 MSW 拦截正常、openapi.yaml 三人确认锁定。这一天没做到，次日全员补救。

D3-D6 是主干并行，三条线同时推进：后端的出价核心和 WebSocket 网关，前端的直播间页和商家后台。

**D7 和 D10 是两个联调日**，这是我们风险收敛的关键闸口。每次联调日都有完整的测试清单，跑完才算过。

D9 合同冻结，之后只接受 P0 级改动。这个时间点非常重要——它意味着后期的稳定性由机制保证，不靠自觉。

D12 是机动日加彩排，不允许新功能，只做 bug fix 和演练。

## 幻灯片 5｜系统全局架构

好，现在进入系统本身。

我们有三个端：移动端 H5 给买家竞拍用，PC 商家后台给卖家发布管理，后端是一套 Go + Gin 的单体模块化服务，数据层是 MySQL 8 加 Redis 7。

架构上我们选择了**单体模块化分层**，没有引入 Kratos、go-zero 这类重型微服务框架。原因很直接：本课题的核心难点是出价一致性和实时同步，微服务会增加大量样板代码，但对解决这两个问题没有帮助。

后端的分层是：Transport 负责鉴权和参数绑定，Application Service 负责事务编排，Domain Rules 负责可单元测试的竞拍规则，Repository 负责 MySQL 和 Redis 的访问。核心出价事务保留明确的手写 SQL，不用 ORM 隐藏条件更新语义。

## 幻灯片 6｜一笔出价，五道防线

出价是整个系统最核心、也是最难的部分。直播间里可能有几百人同时点"出价"，我们需要保证最终只有一笔有效，价格单调递增，不出现重复订单。

我们设计了五道防线：

**第一道，Redis 锁。** key 是 `bid_lock:{auctionId}`，TTL 3 秒，用来削峰，降低数据库的并发压力。注意：Redis 锁只用来削峰，不能作为正确性的唯一来源。

**第二道，MySQL 条件更新。** 这才是真正的正确性保障。UPDATE 语句带上 `AND status='active' AND end_time > NOW() AND current_price + price_step <= ? AND ceiling_price IS NULL OR ? <= ceiling_price`。影响行数为 0 就说明出价已失效，直接返回竞争失败。

**第三道，幂等表。** 同一个 Idempotency-Key 的请求，无论来几次，只处理一次，后续重复返回第一次的结果。这解决了移动端弱网重试的问题。

**第四道，数据库唯一约束。** bids 表有唯一约束防重复出价，orders 表的 auction_id 唯一约束防止同一场拍卖建两张订单。

**第五道，outbox 模式。** 出价记录写入和事件发布在同一个事务里，状态变更和 WebSocket 推送强一致，不会出现"价格涨了但推送没发出去"的情况。

压测目标：单房间 200 QPS 出价，P95 < 200ms，最终价格单调递增。

## 幻灯片 7｜弱网环境下的状态一致性

移动端用户会断网、切换 4G/WiFi、把 APP 切到后台，这些场景都会导致 WebSocket 断开。我们需要保证用户回来的时候，看到的价格和状态是准确的。

我们的设计核心是**事件序号**。服务端每个事件都有全局单调递增的 seq，客户端按 seq 去重和排序。收到比本地 last_seq + 1 更大的事件，说明中间有遗漏，触发补偿。

断线恢复分三级：

第一级，断线后**立即调 `/status` 刷一次快照**，先把当前状态同步过来，不让用户看到过期价格。

第二级，重连时携带本地 `last_seq`，服务端尝试补发缺失的事件。这样重连后不需要完整刷新，只补差量。

第三级，如果 15 秒内 WebSocket 始终恢复不了，**降级为每 2 秒短轮询 `/status`**，保证用户至少能看到接近实时的状态。

另外，页面从后台切回前台时，也必须主动调 `/status` 重新校准，不能相信本地缓存。

## 幻灯片 8｜事件不丢失，状态不重复

拍卖的生命周期由一个后台 Worker 管理：定时扫描 pending 状态的拍卖，到时间就开始；扫描 active 状态的拍卖，时间到了就结束，并且创建订单。

这里有一个经典的分布式问题：多个服务实例同时运行 Worker，怎么保证拍卖不被开始两次、订单不被建两次？

我们的方案是：**不依赖单进程内存 ticker，而是通过 MySQL 条件更新抢占**。开始拍卖的 SQL 带上 `WHERE status='pending' AND start_time <= NOW()`，多个实例同时执行，数据库保证只有一个能 UPDATE 成功。结束拍卖时，在事务内锁定 auction 行，确认状态后才插入订单，`orders.auction_id` 的唯一约束作为最后一道保险。

outbox 表记录所有待发布的事件，publisher 扫描并投递到 WebSocket 房间后标记为 published。允许重复投递，客户端按 seq 去重。

## 幻灯片 9｜一套合同，三端共生

三个端能在 13 天内做到可以联调，靠的是这套接口设计。

几个值得关注的设计决策：

**错误码体系**：HTTP 状态码表达协议层结果，`code` 字段表达业务结果。例如出价低于最低有效价，HTTP 返回 200，code 返回 2101，同时 data 里带上 `min_acceptable_amount`，前端直接预填到出价输入框。

**幂等 Key**：出价和支付接口必须携带 `Idempotency-Key` header，格式约定为 `bid-{userId}-{auctionId}-{timestamp}`。客户端生成，服务端持久化 24 小时。

**封顶价自动成交**：当出价等于封顶价时，在同一个事务里：置拍卖状态为 ended、插入订单、写 outbox BidAccepted + AuctionEnded。HTTP 响应里增加 `ceiling_hit: true` 和 `order_id`，前端无需等 WebSocket，直接切到成交状态。

前端三端共享 `api-client`、`auth`、`time`、`types` 四个 lib，避免同一逻辑写三遍。

## 幻灯片 10｜我们学到了什么

最后做个总结。

**技术层面**：出价一致性不能只靠 Redis 锁，必须多层次叠加。Redis 锁是优化，MySQL 条件更新是正确性，幂等表是用户体验，唯一约束是最后兜底，缺一不可。

**工程层面**：接口合同是并行开发的地基。合同锁定得越早，后期的沟通成本和返工就越少。我们把最多的时间花在合同阶段，而不是开发阶段，这是我们能在 13 天完成的核心原因。

**AI 协作层面**：AI 最大的价值不是"写代码很快"，而是帮你在开始动手之前，把模糊的设计变成精确的合同、把口头的约定变成可执行的文档。一旦合同清晰，AI 生成的每一段代码都是可以被验证的。

一句话总结：**Vibecoding 不是让 AI 替你思考，而是用 AI 把你的思考效率放大 10 倍。**

## Q&A 参考答案

**Q：如果 Redis 挂了，出价还能用吗？**

A：可以降级。Redis 锁本来只是削峰，不是正确性来源。Redis 不可用时，我们的方案是关掉 Redis 锁这一层，直接走 MySQL 条件更新，同时收紧限流（出价从 3 次/秒降到 1 次/秒）。正确性由 MySQL 的条件更新和唯一约束保证，这两个在 Redis 挂了之后仍然有效。

**Q：WebSocket 广播延迟怎么量化？**

A：我们在 outbox 表记录事件创建时间，在广播时记录 published_at，两者之差就是 `ws_broadcast_latency_ms`。目标是 P95 < 500ms。压测时我们用 k6 模拟 1000 个并发 WS 连接，观察这个指标。

**Q：13 天够吗？有什么功能被砍掉了？**

A：主链路全部实现了。预先定义了红线：D6 EOD 如果主链路不通，砍 Task J 压测、砍 PC 监控页、砍移动端订单页。实际上我们主链路在 D6 跑通了，所以这些功能都保留了。被简化的是：音效/触感动画只做了基础版，直播流用占位视频，没有接真实的 CDN。

**Q：为什么不用消息队列（Kafka/RocketMQ）做事件推送？**

A：这是有意的取舍。本课题是单体模块化服务，引入消息队列会带来额外的运维复杂度和部署依赖。outbox 模式足以满足我们的一致性要求，publisher 是一个轻量的 goroutine，失败保留重试。如果要扩展到多机房或者更大规模，把 outbox publisher 替换成 Kafka producer 是很自然的升级路径，合同层不需要变。
