## 板块一：基于 Vibecoding 的 AI 协作开发流程（4-5 张）

这个板块的核心叙事是：我们如何用 AI 驱动的方式在 13 天内完成一个本来要 4-6 周的项目，
体现的不是"用了 AI"，而是一套有方法论的工程思路。

### 幻灯片 1 - 开篇/背景
标题：13 天，三个人，一套生产级竞拍系统
- 简单交代项目背景、时间约束（答辩截止 6 月 10 日）、团队规模（3人）
- 抛出核心问题：为什么这在传统开发模式下很难完成？

### 幻灯片 2 - 我们的 AI 协作方法论
标题：合同先行 × Mock 优先 × 角色专精
核心思路：
- 第一步"锁合同"：在开发前，三人用 AI 协作将接口合同（contract-v2.md）、
  数据库 schema、OpenAPI spec 全部锁定，这是并行开发的地基
- 第二步"解耦并行"：通过 MSW (Mock Service Worker) 让前端不等后端，
  通过 MOCK_MODE 让后端各模块不互相阻塞，D1 第一天三人真正并行
- 第三步"模块级切真"：后端模块完成 → @前端 → 撤 mock → 验收，
  杜绝 mock 漂移，联调可预测
- 用一张流程图展示：合同锁定 → 各自进入 mock 开发 → 模块级切真接口 → 联调日全链路验收

### 幻灯片 3 - AI 在这套流程中的具体角色
标题：AI 做什么，人做什么
用对比表格呈现：
- AI 负责：生成 openapi.yaml 初稿 / MSW handler 代码 / 重复模式的 CRUD /
  错误码映射 / 压测脚本 / 文档整理
- 人负责：出价一致性的架构决策 / 跨团队依赖的协调 / 联调时 bug 根因分析 /
  性能 tradeoff 判断 / 演示顺序设计
- 核心洞察：AI 帮你把"能自动化的重复思考"变成 0，让工程师可以把认知资源
  全部投入到不可自动化的决策上

### 幻灯片 4 - 13 天里程碑节奏
标题：有节奏的冲刺，不是混乱的赶工
- 展示 5 个阶段的时间线（Phase 1 底座 → Phase 2 主干并行 → Phase 3 联调
  → Phase 4 收尾 → Phase 5 答辩）
- 强调两个联调日（D7、D10）是风险收敛的关键闸口
- 强调合同冻结（D9）作为后期稳定性保证
- 体现 Vibecoding 的核心理念：把不确定性前移，用节奏管控风险

## 板块二：系统设计（5-6 张）

这个板块的叙事是：这套系统在技术层面解决了哪些真实的工程难题，做了什么取舍。

### 幻灯片 5 - 系统全局架构
标题：单体模块化分层，不是微服务
- 架构图：移动端H5 + PC商家后台 → HTTP/WebSocket → Go后端（Gin）
  → MySQL 8 + Redis 7
- 后端模块分层：Transport → Application Service → Domain Rules → Repository
- 强调为什么不用微服务框架（Kratos/go-zero）：课题目标是出价一致性和实时同步，
  微服务增加样板代码但不解决核心问题

### 幻灯片 6 - 核心难题：高并发出价一致性
标题：一笔出价，五道防线
- 用流程图展示出价链路的五个保障层：
  1. Redis 锁（bid_lock:{auctionId}，削峰）
  2. MySQL 条件更新（带 AND status/price/ceiling 约束，兜底正确性）
  3. 幂等表（防重复提交和移动端重试）
  4. 数据库唯一约束（bid + order 两张表）
  5. outbox 模式（DB 状态和事件发布强一致）
- 展示核心 SQL（UPDATE ... WHERE ... AND current_price + price_step <= ?）
- 压测目标：单房间 200 QPS 出价，P95 < 200ms，最终价格单调递增

### 幻灯片 7 - WebSocket 实时同步与弱网恢复
标题：弱网环境下的状态一致性
- 展示事件序号机制：seq 单调递增，客户端按 seq 去重和排序
- 三级恢复策略：
  1. 断线 → 立即调 /status 刷快照
  2. 重连携带 last_seq → 服务端补发缺失事件 /events
  3. 15s 未恢复 → 降级短轮询 /status（每2秒）
- 体现移动端场景的特殊性：页面后台切换必须重新校准状态

### 幻灯片 8 - 拍卖状态机与 Outbox 一致性
标题：事件不丢失，状态不重复
- 展示拍卖状态机（pending → active → ended / cancelled）
- 展示 Lifecycle Worker 的多实例安全设计：
  不依赖单进程 ticker，通过 MySQL 条件更新抢占，orders.auction_id 唯一约束防重复建单
- 展示 outbox 表结构和 publisher 流程：扫描 pending → 投递 WS → 标记 published
- 强调核心设计原则：HTTP 写入可能成功，事件投递最终一致，客户端按序号自愈

### 幻灯片 9 - 多端架构与接口设计亮点
标题：一套合同，三端共生
- 三端的分工和复用策略（api-client / auth / time / types 前端共享 lib）
- 展示错误码体系（业务码 vs HTTP 状态码分离）
- 展示幂等 Key 机制（Idempotency-Key header，出价 + 支付必填）
- 封顶价自动成交的特殊处理（同一事务内结束拍卖 + 建单 + 写 outbox）

### 幻灯片 10 - 总结与思考
标题：我们学到了什么
- 技术层面：出价一致性不能只靠 Redis 锁，必须多层次叠加
- 工程层面：接口合同是并行开发的地基，锁得越早，后期风险越小
- AI 协作层面：AI 最大的价值不是写代码快，而是帮你提前把模糊的设计变成可执行的合同
- 一句话总结：Vibecoding 不是让 AI 替你思考，而是用 AI 把你的思考效率放大 10 倍
